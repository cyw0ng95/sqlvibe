## Recent Updates (2026-02-20) - Round 3

Improvements raising pass rate from 49/56 (87.5%) to 54/56 (96.4%):

- **E111** (now passing): Window functions implemented — `COUNT(*) OVER ()`, `COUNT(*) OVER (PARTITION BY key)`, `LAG`, `LEAD`, `FIRST_VALUE`, `LAST_VALUE`, `ROW_NUMBER`, `RANK`, `DENSE_RANK` with `PARTITION BY` and `ORDER BY`. Default frame `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` applied to `LAST_VALUE`. Partition-sorted output matches SQLite row order.
- **F041** (now passing): Fixed two remaining JOIN issues:
  - MixedJoinTypes: LEFT JOIN + INNER JOIN null row emission now scans the INNER JOIN table before emitting the null-left-join row
  - MultipleJoins: `TableColSources` now uses table aliases for correct column-to-cursor assignment in self-joins
- **F201** (now passing): Fixed two remaining CAST issues:
  - CastBlobToText: `Query()` now routes DML statements (INSERT/UPDATE/DELETE) to `Exec()` so INSERT called via Query() (test helper pattern) is actually executed
  - CastInGroupBy: GROUP BY key sort is now NULL-first and numeric-aware, matching SQLite's group ordering behavior
- **pkg/sqlvibe SumMixed**: Fixed pre-existing bug where `extractAggregatesFromExpr` de-duplicated aggregates by function name only (causing `SUM(i) + SUM(r)` to return 20 instead of 13.14). Now de-duplicates by function-name+args key.

### Remaining Priority Gaps

1. **E041 SelectFromT1UsingIndex**: Row ordering without ORDER BY — SQLite uses a DESC index scan and returns rows in reverse order; sqlvibe returns insertion order. Both orderings are valid per SQL standard. Fix: add ORDER BY to the test or implement index-based scan ordering.
2. **E101 Complex UPDATEs**: `UpdateOrderBy` with `WHERE id IN (SELECT id FROM ... ORDER BY salary DESC LIMIT 2)` — LIMIT in the IN-subquery may not be correctly applied, causing extra rows to get `salary + 100`. Also `SelectNullBonus` — NULL bonus set by `UpdateWithNull` may be overwritten by a subsequent correlated subquery UPDATE.
